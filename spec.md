# AI Chatbox Web MVP 需求大纲

## 1. 项目概述

本项目目标是从零设计并实现一个 Web 版 AI Chatbox，整体交互参考 ChatGPT，优先完成可演示、可联调、可继续扩展的 MVP 版本。

当前阶段以“演示型 MVP”为定位，要求具备完整的基础对话闭环，同时在架构上预留后续扩展能力，避免一次性写死。

## 2. 项目目标

- 提供一个可在浏览器中直接使用的 AI 聊天页面。
- 支持基础多轮对话能力。
- 支持 AI 回复的流式输出体验。
- 支持多会话管理和切换。
- 支持浏览器本地保存聊天记录，刷新后可恢复。
- 提供 mock 测试 API 和演示 Key，便于前后端联调。
- 支持接入 OpenAI 兼容接口，便于后续切换真实模型服务。
- 保证桌面端和移动端都具备基础可用性。

## 3. 当前范围

### 3.1 本期包含

- 聊天主页面与基础布局。
- 输入框、消息列表、消息气泡、发送按钮等核心交互。
- 会话列表、新建会话、切换会话。
- 用户消息即时展示与助手消息流式渲染。
- 基础错误处理和重试能力。
- 浏览器本地存储会话与消息。
- 后端聊天接口、测试接口、健康检查接口。
- mock provider 与 OpenAI-compatible provider。
- 基础单元测试与端到端测试。
- README 和环境变量说明。

### 3.2 本期不包含

- 登录、注册、用户体系。
- 服务端数据库持久化。
- 文件上传、图片理解、语音能力。
- 联网搜索、插件系统、工具调用。
- 计费、权限控制、审计日志。
- 生产级监控、风控和高可用部署方案。

## 4. 用户体验要求

- 页面结构应接近主流 AI 对话产品，具备清晰的会话区、消息区和输入区。
- 用户发送消息后，消息需立即出现在消息列表中。
- 助手回复需采用流式或分段增量渲染，而不是等待完整结果后一次性展示。
- 输入框支持回车发送、Shift+Enter 换行。
- 发送中需要有明确状态反馈，并避免重复提交。
- 当接口失败、超时或配置异常时，界面需显示明确错误提示，并允许用户重试。
- 移动端需要可以打开、关闭和切换会话列表，避免遮挡核心输入区域。

## 5. 核心功能模块

### 5.1 聊天页面模块

- 展示当前会话的全部消息。
- 区分用户消息和助手消息的视觉样式。
- 支持 Markdown 文本显示。
- 支持代码块展示。

### 5.2 会话管理模块

- 支持新建会话。
- 支持在多个会话之间切换。
- 基于首条用户消息自动生成会话标题。
- 支持页面刷新后恢复历史会话。

### 5.3 本地存储模块

- 使用浏览器本地存储保存会话与消息。
- 使用统一数据结构，方便后续迁移到服务端存储。
- 为未来的数据版本升级保留 schema version 扩展位。

### 5.4 AI Provider 模块

- 提供统一的 provider 抽象层，屏蔽具体模型服务差异。
- 支持 mock provider，用于开发和测试。
- 支持 OpenAI-compatible provider，用于对接真实模型服务。

### 5.5 API 接口模块

- `POST /api/chat`：正式聊天接口。
- `POST /api/test/chat`：固定走 mock provider 的测试接口。
- `GET /api/health`：返回服务当前 provider、模型与可用状态。

## 6. 技术方案

- 前端与后端框架：Next.js + React + TypeScript。
- 样式方案：Tailwind CSS。
- 服务端接口层：Next.js Route Handlers。
- 数据校验：Zod。
- 单元测试：Vitest。
- 端到端测试：Playwright。
- 模型接入方式：
  - `mock` 模式：用于本地测试和演示。
  - `openai-compatible` 模式：通过服务端环境变量接入兼容 OpenAI 协议的模型服务。

## 7. 数据模型与接口约定

### 7.1 数据模型

- `ChatConversation`
  - `id`
  - `title`
  - `createdAt`
  - `updatedAt`
- `ChatMessage`
  - `id`
  - `conversationId`
  - `role`
  - `content`
  - `createdAt`
  - `status`
  - `error?`

### 7.2 接口约定

- `POST /api/chat`
  - 入参：`conversationId`、`messages`、可选 `model`、可选 `systemPrompt`
  - 返回：流式聊天响应
- `POST /api/test/chat`
  - 入参与 `/api/chat` 保持一致
  - 认证：`Authorization: Bearer demo-chat-key`
  - 返回：mock 流式聊天响应
- `GET /api/health`
  - 返回：当前 provider、模型和服务状态

## 8. 非功能要求

- 真实 API Key 仅允许保存在服务端环境变量中，不得暴露到前端。
- 代码结构应清晰，模块边界明确，方便后续扩展。
- 核心操作失败时必须给出明确反馈，避免静默失败。
- 页面在主流桌面和移动浏览器下应保持基础可用。
- README 应足以指导开发者完成本地运行、环境配置和测试接口联调。

## 9. 阶段里程碑

### 阶段 1：项目初始化与工程骨架

- 建立前端、后端、样式和测试基础设施。

### 阶段 2：核心聊天能力

- 打通输入、发送、流式回复、错误处理和基础 UI。

### 阶段 3：会话管理与本地存储

- 完成多会话、标题生成和页面刷新恢复。

### 阶段 4：测试与交付文档

- 补齐测试、README、任务清单和验收标准。

## 10. 后续扩展方向

- 服务端持久化与用户体系。
- 会话重命名、删除与归档。
- 模型切换与系统提示词设置。
- 文件上传、多模态能力。
- 联网搜索、工具调用与插件能力。
